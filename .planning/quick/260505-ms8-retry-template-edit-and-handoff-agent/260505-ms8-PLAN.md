# Retry dialog: editar template + escolher agente

## Solicitação do usuário

> "Na tela de reenvio que abra o template da campanha original e que eu possa escolher o agente para dar continuidade. Tem um pequeno erro na primeira mensagem, preciso corrigir. Mas nada de estragar o fluxo."

## Decisão

3 mudanças, todas opcionais. Quem não tocar nos novos campos vê comportamento idêntico:

1. **Preview endpoint** `POST /conversations/retry-failed/preview` retorna template(s) das campanhas das conversas selecionadas + agentes inbound ativos. Não muta nada.
2. **`retrySchema`** ganha 2 campos opcionais:
   - `messageOverrideTemplate?: string` — texto da mensagem (com variáveis); se ausente, usa `messages.content` original (comportamento antigo).
   - `handoffAgentId?: string` — UUID de agente inbound; se ausente, mantém `conversations.agentId` atual.
3. **RetryModal** abre o template, oferece checkbox "editar mensagem", textarea, e dropdown de agentes.

## Detalhes do backend (`apps/api/src/routes/conversations.ts`)

- Preview faz `SELECT DISTINCT campaign + template + agentes inbound`. Se todas as convs compartilham 1 campanha + template, retorna `uniformTemplate`. Caso contrário, retorna lista por campanha + null em uniformTemplate (frontend mostra warning).
- Validação inline: `handoffAgentId` precisa ser de `agents.type='inbound' AND active=true`. Se inválido, badRequest antes de qualquer escrita.
- Render: nova função local `renderRetryTemplate` espelha exatamente `renderTemplate` + `VAR_ALIASES` de `packages/agent-engine/src/run.ts:74-114` (duplicação consciente — evita exportar internals do engine).
- Per-conv: SELECT DISTINCT ON expandida com `INNER JOIN leads + LEFT JOIN campaigns` pra trazer `lead.name/phone/propertyRef/origin/campaign.name`. Usa essas vars no render quando `messageOverrideTemplate` está set.
- Bulk update de `conversations.agentId` em UMA query `WHERE id IN (...)` quando handoffAgentId é set.

## Detalhes do frontend (`apps/web/components/chat/ConversationListPane.tsx`)

- `previewQuery` chama o novo endpoint na abertura do modal.
- 2 useState novos: `editTemplate` (boolean) + `overrideTemplate` (string) + `handoffAgentId` (string).
- useEffect pré-preenche `overrideTemplate` com `uniformTemplate` na primeira load — não clobbera edições do user em re-fetches.
- Textarea só aparece com `editTemplate=true`. Caso contrário, mostra um `<pre>` read-only do template.
- Dropdown de agentes mostra `Manter agente atual` como opção padrão (vazia).
- Submit envia os campos novos só quando preenchidos (spread condicional `...(condition ? { x } : {})`).

## Trade-offs aceitos

- **Duplicação do `renderTemplate`** — refatorar pra exportar do `@pointer/agent-engine` mexeria no engine que está estável. Custo de 30 linhas duplicadas é menor que risco de regressão.
- **Override é template, não texto literal** — manter substituição de variáveis preserva personalização per-lead (vital com 964 retries). Custo: usuário precisa entender as chaves. Mitigado por hint visível com aliases PT-BR.
- **Sem opção "salvar template corrigido na campanha"** — usuário pode editar manualmente em `/app/campaigns` se quiser persistir. Adicionar o botão aqui aumenta surface de erro (ex: salvar acidentalmente). Pode virar quick task se virar pedido.
- **Sem busca/filtro no agent dropdown** — assume poucos agentes inbound. Se virar 50+, vira ux problem.
- **Multi-campaign warning** mas funcional — dá pra reenviar com 1 texto pra convs de campanhas diferentes. Render usa as vars de cada lead, então é seguro.

**Files:** `apps/api/src/routes/conversations.ts`

- Validação handoffAgentId como inbound active.
- Expandir DISTINCT ON da retry-failed pra trazer `lead.name/phone/propertyRef/origin + campaign.name` via JOINs.

**Verify:** `pnpm --filter @pointer/api exec tsc --noEmit` → 0 erros. Comportamento sem novos campos é idêntico ao antes (mesmo fluxo de DISTINCT ON + bulk insert + bulk addBulk).

**Files:** `apps/web/components/chat/ConversationListPane.tsx`

**Action:**

**Verify:** `pnpm --filter @pointer/web exec tsc --noEmit` → 0 erros.

**Done:** Operador vê template, edita se quiser, escolhe agente, ou simplesmente não toca em nada e segue como antes.

## Verificação manual (após deploy)

1. Inbox > filtro Falhadas > seleciona 1-2 conversas > Reenviar.
2. Esperado: dialog mostra template da campanha, agentes no dropdown.
3. **Cenário A (não tocar em nada):** clica Confirmar. Esperado: comportamento atual — reenvia conteúdo original.
4. **Cenário B (editar template):** marca checkbox, corrige typo, clica Confirmar. Esperado: cada conv recebe o texto novo com `{{name}}` substituído pelo nome do lead.
5. **Cenário C (escolher agente):** seleciona agente inbound no dropdown, Confirmar. Esperado: `conversations.agentId` atualizado em batch; quando lead responder, o agente escolhido pega.

## Out of scope

- Botão "Salvar correção no template da campanha" (persistir em `campaigns.first_message_template`) — separado, se virar dor.
- Preview de mensagem renderizada para 1 lead — separado.
- Anexos extras no retry (vídeo, imagem) — out.
- Anexar agentes outbound também — out (continuidade só faz sentido com inbound).
