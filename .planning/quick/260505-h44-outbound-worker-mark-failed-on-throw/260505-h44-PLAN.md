# Worker: marcar message failed quando BullMQ esgota retries

## Sintoma

Após `b99b51d` (filtro UI), msgs em `status='queued'` ficam invisíveis no chat. User clica retry, dialog confirma "Agendado: 1 mensagem(ns)", mas:
- Mesmo com instâncias estáveis, msg não aparece como sent.
- Msg também não aparece como failed (não vai pro filtro "Falhadas").

User: "ainda fica em fila e nao esta enviando mesmo as instancias estando estaveis".

## Raiz

`apps/worker/src/jobs/outbound-message.ts` tem 2 branches de throw que NÃO mexem no DB:

1. `campaign_rate_limit` (linhas 65-77): lança `Error("campaign_rate_limit")` quando `reserveCampaignSlot` retorna waitMs.
2. `no_instance_available` (linha 184-187): lança quando picker não acha instance.

Comportamento BullMQ:
- Throw → BullMQ marca attempt failed → retry com backoff exponencial.
- Default `attempts: 3, backoff: { type: "exponential", delay: 2000 }` → ~6s total.
- Após 3 tentativas, JOB vai pra failed queue do BullMQ.
- Mas `messages.status` continua **queued para sempre**.

Resultado: msg fica num limbo. Não está no filtro "Falhadas" (só pega status='failed'), não dá pra retry pela UI, não tem rastro de erro no inbox.

## Decisão

3 fixes:

1. **Marcar failed antes do throw** nas 2 branches.
2. **Safety-net failed handler** no worker pra cobrir qualquer outro path terminal (ex: erro DB, lib bug).
3. (já feito em quick task anterior `260505-g83`) UI filter — só mostra outbound queued depois que worker resolve.

### Por que marcar failed e ainda assim throw?

Para preservar o retry do BullMQ quando o ambiente recupera (rate-limit reseta, instância volta online):

- 1ª tentativa: throw → marca failed no DB → BullMQ retry.
- 2ª tentativa: worker reentra. Não tem skip de "failed" (linha 54: só skipa "sent"). Reentra picker.
  - Se acha instance agora → provider.send → sucesso → marca sent (sobrescreve failed). ✓
  - Se não acha → marca failed de novo (idempotente) → throw → retry.
- 3ª tentativa: igual.
- Após esgotar attempts: status fica failed permanentemente. Inbox mostra failed. User pode retry-failed.

### Por que safety-net no failed listener também?

Cobre paths não previstos. Ex: `db.query.conversations.findFirst` lança porque DB caiu, ou import de provider falha. O try/catch principal (linhas 203-266) cobre só o bloco do provider.send. Anything que lança ANTES disso passa direto pro BullMQ retry sem mexer no DB.

Verificação na listener: `attemptsMade >= opts.attempts ?? 3`. Só intervém na última tentativa para preservar retry intermediário.

## Trade-offs aceitos

- **Status flap durante retries**: msg pode ficar `failed` brevemente entre attempts. Cliente vendo em tempo real pode ter UX confusa. Aceito porque (a) é raro um usuário olhar o exato instante de retry, (b) o sucesso final sobrescreve para `sent`, (c) corrigir requer rastrear `attemptsMade` dentro do handler — tradeoff complexidade × benefício marginal.
- **Não rastreia última razão do erro**: schema de `messages` não tem `last_error`. Adicionar coluna é fora de escopo aqui. Logs do worker mostram (`outbound: campaign rate limit hit`, `outbound: no available instance`) — investigação fica via logs.
- **Safety-net duplica intent**: tanto as branches inline quanto o listener marcam failed. Listener só dispara em paths não cobertos pelas branches. Aceito como defesa em camadas.
