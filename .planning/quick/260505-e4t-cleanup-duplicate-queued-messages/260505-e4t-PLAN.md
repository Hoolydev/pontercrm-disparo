# Cleanup script — duplicate queued messages

## Sintoma

Inbox mostrando várias mensagens idênticas (mesmo conteúdo) na mesma conversa, todas com `status='queued'`, sem nada saindo no WhatsApp.

## Raiz

Antes do fix `b0ede3e`, `/conversations/retry-failed` rodava loop sequencial e estourava gateway timeout em batches grandes (964 convs). O usuário clicou "Reenviar" várias vezes — cada clique inseria messages + enfileirava jobs antes do timeout fechar a request. Resultado: mesma `(conversation_id, content_hash)` ficou com 3+ rows queued.

Com o fix de batch agora as duplicatas não aparecem mais, mas as antigas continuam no DB.

## Decisão

Script standalone em `apps/api/src/cleanup_duplicate_queued.ts` no padrão dos demais ops scripts (`fix_campaign.ts`, `ensure_agents.ts`):

1. SQL com `ROW_NUMBER() OVER (PARTITION BY conversation_id, content_hash ORDER BY created_at ASC)` — mantém a mais antiga, marca as outras como duplicatas.
2. Filtro: `direction='out'`, `status='queued'`, `content_hash IS NOT NULL`.
3. Dry-run default — só conta + amostra. `--execute` deleta.
4. Best-effort: pra cada msg duplicada, tenta `queues.outboundMessage.getJob(jobId)` em prefixos conhecidos (`retry-`, `ai-send-`, `prop-`, `camp-att-`) e remove. Se não achar, segue (worker é idempotente — `outbound-message.ts:50-53` já trata `msg not found`).
5. Bulk DELETE em chunks de 1000 ids (mantém parâmetros bounded).

## Trade-offs aceitos

- **Não cobre status='sent' falsos** — usuário mencionou "mostrando como enviada mas não saiu pelo WhatsApp". Se alguma row marcada `sent` realmente não foi entregue, é outro bug (provavelmente provider devolveu 200 mas a instância estava off — id de fallback `uazapi-${Date.now()}`). Investigar separado se aparecer queixa.
- **Best-effort em remover BullMQ job** — se não encontrar pelo prefixo, segue. Worker já trata orphan job (DB row some → no-op). Trade: pode "queimar" turno de rate-limit ao processar job órfão.
- **Sem soft-delete / audit log** — DELETE direto. Não há histórico de quais rows foram removidas além do log no stdout.
- **Roda local apontando pra prod** — mesmo padrão dos demais `fix_*.ts`. Não vai pro Railway.

## Verificação manual (user)

1. Setar env locais:
   ```bash
   export DATABASE_URL='postgres://...'  # Neon prod
   export REDIS_URL='redis://...'        # Railway Redis prod
   ```
2. Dry run primeiro:
   ```bash
   pnpm --filter @pointer/api exec tsx src/cleanup_duplicate_queued.ts
   ```
   Deve listar quantas duplicatas existem + amostra de 15.
3. Confirmar que os números fazem sentido. Se sim:
   ```bash
   pnpm --filter @pointer/api exec tsx src/cleanup_duplicate_queued.ts --execute
   ```
4. Verificar no inbox: a conversa do "Marcelo" (e outras) deve ficar com apenas 1 versão da mensagem por conteúdo.

## Out of scope

- Investigar messages com `status='sent'` que não saíram no WhatsApp — separado.
- UI de cleanup — script CLI é suficiente.
- Soft-delete / arquivamento — DELETE direto.
- Constraint UNIQUE em `(conversation_id, content_hash, status)` para prevenir reincidência — não é o caminho certo (legítimo retry pode reenviar mesmo conteúdo). A prevenção real foi o fix de batch (`b0ede3e`).
