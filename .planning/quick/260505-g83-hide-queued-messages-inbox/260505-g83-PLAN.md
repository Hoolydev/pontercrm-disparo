# Hide queued outbound messages from inbox views

## Sintoma

Mensagens com `status='queued'` ainda não dispatched (esperando worker) apareciam no inbox como se tivessem sido enviadas pro WhatsApp. Confundia operador — visualmente "saiu" mas o lead não recebeu.

## Decisão

UX explícita do usuário: msg outbound só deve aparecer no chat quando ela transitiona para `sent`/`delivered`/`read` ou `failed`. Enquanto ela está `queued` (esperando worker dispatch), fica oculta.

Aplicar filtro no backend nos 2 sites:

1. `GET /conversations` — preview da última mensagem na lista lateral.
2. `GET /conversations/:id` — histórico completo no painel de chat.

Filtro: `NOT (direction = 'out' AND status = 'queued')`. Inbound continua aparecendo em qualquer status. Outbound aparece se já transitou (sent/delivered/read/failed).

## Tasks

### Task 1: Filtro no relation `messages` em ambos endpoints

**Files:** `apps/api/src/routes/conversations.ts`

**Action:** Adicionar `where: (m, { not, and, eq }) => not(and(eq(m.direction, "out"), eq(m.status, "queued"))!)` no `with.messages` de ambos os queries.

**Verify:** `pnpm --filter @pointer/api exec tsc --noEmit` → 0.

**Done:** Mensagens queued não aparecem no chat até o worker resolver pra sent ou failed.

## Trade-offs aceitos

- **Latência visual de 0.5-2s no broker manual send**: o broker digita, clica enviar, msg fica invisível até o worker dispatchar. Aceitável porque o broker-send job não tem `delay`, BullMQ pega imediatamente. Se virar dor: adicionar optimistic update no frontend.
- **Latência longa em retry-failed**: msgs com `delay: i*delaySeconds*1000` ficam invisíveis 60s+ para `i > 0`. Esperado e desejado pelo user — só aparece quando "saiu".
- **Sem filtro de status='queued' inbound**: inbound também tem rotas que poderiam estar nesse estado durante webhook race, mas nenhum sintoma reportado. Manter inbound visível em qualquer status preserva log completo de recebimento.

## Verificação manual

1. Após deploy: clicar "reenviar" para 1 conv falhada.
2. Esperado: dialog "Agendado: 1 mensagem(ns)" aparece, mas o chat NÃO mostra a msg ainda.
3. Aguardar worker dispatch (≤ 5s tipicamente). Refresh do inbox.
4. Esperado: msg aparece com status `sent` (ou `failed` se provider deu erro).

## Out of scope

- Bug do worker que pode estar deixando msgs presas em `queued` para sempre — investigado separado se sintoma persistir após filtro.
- UI de "mensagens em fila" (lista paralela mostrando o que está esperando dispatch) — não pedido.
- Optimistic update no broker manual send — só se virar dor.
