# Summary — Hide queued messages from inbox

## Sintoma

Outbound msgs em `status='queued'` apareciam no chat como se já tivessem sido enviadas pro WhatsApp. Lead ainda não recebeu nada, mas operador via como "saiu".

## Fix

`apps/api/src/routes/conversations.ts` — filtro `NOT (direction='out' AND status='queued')` no `with.messages` de ambos:
- `GET /conversations` (preview da última msg na lista)
- `GET /conversations/:id` (histórico completo)

## Verificação

- `pnpm --filter @pointer/api exec tsc --noEmit` → 0 erros.
- Smoke real precisa do user: após deploy, retry 1 falha → chat NÃO mostra msg até worker resolver.

## Pendência relacionada (separada)

Usuário reportou que retry "cria fila mas não envia". Hipótese: worker não tem instância disponível pra enviar (status `pending` ou `disconnected` no DB). Diagnóstico recomendado:

1. Ir em `/app/instances`.
2. Para cada instância uazapi, clicar **"↻ Verificar conexão"**.
3. Se status real é `connected` mas DB mostra `pending`, o botão sincroniza.
4. Tentar reenviar 1 falha. Se ainda travar `queued > 30s`, abrir nova quick task — provavelmente é `outbound-message.ts` jogando `no_instance_available` em loop, ou provider devolvendo 200 mas lançando exception silente.
