# Menu reorg — Anti-padrão 5 (passo 4 do plano de refactor)

## Contexto

Sidebar atual em `apps/web/app/app/layout.tsx` está organizado por papel (NAV / OPS / ADMIN), não por entidade. Ler o sidebar não dá pista de "onde mexo em X". O plano de refactor previa 4 grupos (CRM/Catálogo/Operação/Configuração) mas auditoria do menu existente (18 itens) mostra que Catálogo fica com 1-2 itens — section header dead weight.

Decisão alinhada com usuário: **3 grupos** (option A).

## Decisões

### Estrutura final

| Section | Role gate | Itens (na ordem) |
|---------|-----------|------------------|
| **CRM** | todos | Dashboard, Inbox, Leads, Campanhas, Agendamentos, Captação |
| **Operação** | supervisor+ | Cobranças, Fila Corretor, Alertas SLA, Métricas |
| **Configuração** | admin | Agentes, Funis, Instâncias, Triggers, Fontes, Usuários |

### Mudanças semânticas

1. **Métricas migra Admin → Operação**. Hoje aparece tanto no sidebar (sob "Admin", visível pra qualquer supervisor) quanto na top bar (visível a todos via Link direto). Migrar:
   - Sidebar: vai pra Operação. Visibilidade efetiva igual à atual (supervisor+).
   - Top bar: link permanece. Brokers continuam acessando via top bar (preserva UX atual — não é regressão).

2. **Fontes (sources) permanece em Configuração** (admin only) — quem criou veio do contexto da decisão (uso raro, parametrização).

3. **CRM ganha header "CRM"** acima dos itens de NAV. Hoje os itens de NAV não têm header (só OPS e ADMIN têm). Adicionar header "CRM" deixa visualmente paralelo com os outros — mais limpo.

4. **Section "Admin" → "Configuração"**. Renomear o label.

5. **Section "Operação"**: label fica.

Mudança fina: o label do `/app/leads` virou "Leads" (era "CRM"). Como a section inteira agora é "CRM", o item "CRM" dentro de CRM ficaria redundante. "Leads" descreve o que a página é.

### Render

Substituir o block atual NAV-first + OPS condicional + ADMIN condicional por 3 sections paralelas, todas com header (incluindo CRM). Pattern uniforme:

```tsx
<NavSection label="CRM" items={CRM_NAV} collapsed={collapsed} pathname={pathname} />
{isSupervisor && (
  <NavSection label="Operação" items={OPS_NAV} collapsed={collapsed} pathname={pathname} />
)}
{isAdmin && (
  <NavSection label="Configuração" items={CONFIG_NAV} collapsed={collapsed} pathname={pathname} />
)}
```

Extrair pequeno componente `NavSection` (interno ao arquivo) para evitar repetir o `<div mt-5 pt-4 border-t><p>HEADER</p>{items.map(...)}</div>` 3 vezes. Primeira section (CRM) não tem o `border-t` (é a primeira) — `NavSection` aceita prop `first?: boolean`.

### Top bar

Botão "Métricas" da top bar (linhas 261-274) **permanece intocado**. Brokers continuam acessando via top bar quando precisam (não tinham sidebar pra isso). Razão: ainda há uma diferença de visibilidade que vale auditar separadamente, mas não é o escopo desta task.

## Trade-offs aceitos

- **Brokers ainda acessam Métricas via top bar**: o link no `<header>` é independente do sidebar, e remover quebraria UX atual (rola atrás depois se for política nova). Aceito como pré-existente, fora do escopo.
- **Label "Leads" em vez de "CRM"**: ligeiro retrabalho de muscle memory pra quem usa o sistema (clica em "CRM" hoje), mas a clareza ganha sobrepõe — section "CRM" + item "Leads" deixa óbvio "vai em CRM > Leads".
- **Header também em CRM**: hoje a primeira section não tem header. Adicionar deixa parallel structure. Trade: 12px de altura a mais no topo do sidebar.
- **Sem mudança de role gates**: a expectativa "Configuração admin only" e "Operação supervisor+" preserva quem viu o que antes. Ninguém perde acesso a página por essa task.
- **Sem novos atalhos / breadcrumbs**: escopo restrito ao sidebar. CommandPalette (`apps/web/components/CommandPalette.tsx`) tem listas próprias — fora do escopo.
