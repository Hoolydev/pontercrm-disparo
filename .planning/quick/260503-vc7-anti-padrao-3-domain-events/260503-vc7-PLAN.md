# Domain events — Anti-padrão 3 (passo 3 do plano de refactor)

## Contexto

Anti-padrão 3 do plano: não há audit log durável de mutações de domínio. Hoje:
- `inbox:updates` Redis pubsub é ephemeral — UI live recebe, ninguém persiste.
- Auditoria é por inferência (lê o estado final + scattered logs).
- Não dá pra reconstituir "quando esse lead virou hot" ou "quem moveu de 'qualificado' pra 'agendado' na quarta-feira".

Decisão de design (alinhada com o usuário antes do plano):

1. Schema: `domain_events` com `aggregate_type`, `aggregate_id`, `event_type`, `payload jsonb`, `actor`, `occurred_at`. Sem `version`/`sequence` (não é event sourcing — é audit log + integração futura).
2. Transacionalidade: services aceitam `tx?: Database` opcional. Default = mesmo `db` do caller. Hoje ninguém passa tx; fica forward-compatible quando alguém precisar de atomicidade real.
3. Eventos iniciais (3): `lead.stage_changed`, `conversation.status_changed`, `broker.assigned`. Followup/scoring/message events ficam para depois.
4. Pubsub Redis coexiste — `inbox:updates` continua intocado para a UI live; `domain_events` é o log durável paralelo.

Este passo NÃO entrega consumers (webhook out, audit endpoint, replay) — só infra de gravação. Próximos passos podem consumir.

## Decisões

### Schema (Drizzle)

`packages/db/src/schema/domain-events.ts`:

```ts
export const domainEvents = pgTable(
  "domain_events",
  {
    id: id(),
    aggregateType: text("aggregate_type").$type<DomainAggregateType>().notNull(),
    aggregateId: uuid("aggregate_id").notNull(),
    eventType: text("event_type").$type<DomainEventType>().notNull(),
    payloadJson: jsonb("payload_json").$type<Record<string, unknown>>().notNull().default({}),
    actor: text("actor"),
    occurredAt: timestamp("occurred_at", { withTimezone: true }).notNull().defaultNow()
  },
  (t) => ({
    aggregateIdx: index("domain_events_aggregate_idx")
      .on(t.aggregateType, t.aggregateId, t.occurredAt),
    eventTypeIdx: index("domain_events_event_type_idx").on(t.eventType, t.occurredAt)
  })
);
```

Tipos em `packages/shared/src/roles.ts`:
- `DOMAIN_AGGREGATE_TYPE = ["lead", "conversation", "broker_queue"] as const`
- `DOMAIN_EVENT_TYPE = ["lead.stage_changed", "conversation.status_changed", "broker.assigned"] as const`

`aggregateId` é `uuid` mesmo quando o aggregate é "broker_queue" (rows de broker_queue têm uuid). Não FK explícita — o aggregate row pode ser deletado/arquivado e queremos preservar histórico (audit log).

### Helper `recordEvent`

`packages/agent-engine/src/lib/domain-events.ts`:

```ts
export interface RecordEventOpts {
  actor?: string | null;
  payload?: Record<string, unknown>;
}

export async function recordEvent(
  db: Database,
  aggregateType: DomainAggregateType,
  aggregateId: string,
  eventType: DomainEventType,
  opts: RecordEventOpts = {}
): Promise<void> {
  await db.insert(schema.domainEvents).values({
    aggregateType,
    aggregateId,
    eventType,
    payloadJson: opts.payload ?? {},
    actor: opts.actor ?? null
  });
}
```

Reexportar de `packages/agent-engine/src/index.ts`.

Não retorna o ID do event nem joga exception — falhas de insert do audit log NÃO devem quebrar a operação principal. Catch → `console.warn` interno (logger não é injetável aqui sem refactor maior). Decisão pragmática: a observabilidade é ganho marginal vs custo de propagar erro ao caller.

Wait — reler. Vou inverter: como o helper roda dentro do mesmo service que mutou a row e o usuário concordou que services aceitam tx opcional para "atomicidade real" futura, o catch silencioso seria armadilha. Se a query falhar agora (ex: schema não migrado), queremos saber em dev. Decisão final: **NÃO catch interno** — propaga. Se algum caller de produção quiser robustez, adiciona try/catch onde chama o service.

### Integração nos services

**`changeLeadStage`** (lib/leads.ts):
- Após o UPDATE bem-sucedido (apenas no branch `changed: true`), chama `recordEvent(db, "lead", leadId, "lead.stage_changed", { actor: opts?.actor, payload: { previousStageId, newStageId } })`.
- Aceita `actor` que já está na assinatura.

**`transitionConversationStatus`** (lib/conversations.ts):
- Após o UPDATE bem-sucedido (apenas no branch `changed: true`), chama `recordEvent(db, "conversation", conversationId, "conversation.status_changed", { payload: { from: conv.status, to: newStatus, handoffReason: patch.handoffReason ?? null, assignedBrokerId: patch.assignedBrokerId ?? null } })`.
- Adicionar `actor?: string | null` ao opts da função (não está hoje — adicionar).

**`recordBrokerAssignment`** (lib/distribution.ts):
- Após o INSERT bem-sucedido (apenas no branch `reused: false`), chama `recordEvent(db, "broker_queue", id, "broker.assigned", { actor: opts.reason ?? null, payload: { leadId, brokerId, conversationId, priorityHint, timeoutAt } })`.

`actor` aqui é null para tudo nesta fase — nenhum caller passa user_id ainda. Ficar pronto pra receber do JWT no anti-padrão 5 (UI reorg) ou em uma quick task posterior.

### O que NÃO entra neste passo

- Consumers (webhook out, audit endpoint, replay).
- Backfill de events para mutações passadas — forward-only.
- Eventos de followup, scoring, message — alto volume, entram com filtragem depois.
- `tx` parameter já mencionado: assinatura aceita, mas nenhum caller passa hoje. Fica como hook.
- UI de visualização (timeline) — anti-padrão 2/5.

## Trade-offs aceitos

- **Sem catch interno em recordEvent**: se DDL não estiver aplicada, requests vão falhar imediatamente com erro Postgres ("relation domain_events does not exist"). Aceito porque (a) é o sinal certo em dev, (b) prod só recebe a build após migration aplicada, (c) catch silencioso é a armadilha clássica de audit log.
- **Sem FK em aggregate_id**: simplifica deletes/arquivamento e suporta múltiplos aggregate types numa coluna. Trade-off: integridade não é forçada pelo banco — bug em código pode escrever ID inválido. Mitigação: code review + testes manuais.
- **`actor` é null em todos os call sites hoje**: aceitável para começar. UI/JWT plumbing entra depois.
- **`recordBrokerAssignment` usa `opts.reason` como actor**: na verdade reason é semântico, não actor. Decisão final: `actor: null`, payload inclui `reason`. Corrigido aqui antes de implementar.
- **Sem versionamento de payload**: schema do payload pode mudar; consumers futuros precisam tolerar evolução. Aceito (audit log, não event sourcing).

## Adendo (revisão antes do execute)

Reler "Trade-offs": vou adicionar `tx?: Database` aos 3 services E ao `recordEvent` na Task 2. Mudança pequena: assinatura vira `(db, …args, opts?: { …, tx?: Database })` ou para simplificar `recordEvent(db | tx, …)` — caller resolve. Decisão: não criar campo separado `tx` em opts. Em vez disso, **services aceitam o primeiro arg `db` que pode ser uma transaction handle do Drizzle** (mesma assinatura `Database`, já é o que `db.transaction(async (tx) => …)` retorna). Helpers atuais (`changeLeadStage`, etc) já aceitam `db` — se um caller futuro abrir transaction e passar `tx` em vez de `db`, funciona sem mudança de API. Conclusão: **nada a fazer** na Task 2 sobre tx — o pattern já está embutido na escolha de receber `db` como primeiro arg. Riscar a pendência do "Out of scope".
